# reference/ 目录

## 定义

**reference/** 是项目根目录下用于集中存储只读外部参考资料的数据结构单元。该目录不属于项目核心源码范畴，其定位明确为"只读参考资料仓库"，用于存放团队在学习、研究和开发过程中引用的外部材料。

在知识体系中的位置：

- 所属领域：[[concept-project-structure]]（项目结构设计）
- 上级概念：项目目录组织规范
- 同级概念：src/ 源码目录、doc/ 文档目录、tests/ 测试目录

该目录的核心特征是"外部性"和"只读性"——存放的是来自项目外部的参考资料，而非项目团队自己编写的代码或文档。这种定位有助于维护项目的关注点分离原则。

## 详细属性

### 基本规格

| 属性 | 值/说明 |
|------|---------|
| 目录名称 | `reference/` |
| 位置 | 项目根目录 |
| 目录性质 | 只读参考资料仓库 |
| 是否包含源码 | 否 |
| 是否纳入版本控制 | 可选（建议大文件使用 LFS 或外链） |

### 内容类型

| 类型 | 说明 | 示例 |
|------|------|------|
| 开源项目 | Clone 的外部开源仓库 | 复现的标杆项目 |
| 学术论文 | PDF 或在线论文链接 | SOTA 算法论文 |
| 技术博客 | 博客文章摘录或全文 | 权威技术教程 |
| 规范文档 | 外部标准或规范 | API 协议规范 |
| 对标材料 | SOTA 技术对比资料 | 竞品分析文档 |

### 存储策略规格

| 文件大小 | 推荐策略 |
|----------|----------|
| < 10MB | 直接存入目录 |
| 10MB - 100MB | 考虑使用 Git LFS |
| > 100MB | 仅保留外部链接 |

## 工作原理

### 目录隔离机制

reference/ 目录的核心设计原理是通过物理目录级别的隔离，实现源码与参考资料的关注点分离。在传统的项目结构中，参考资料往往散落在项目各处，或者与源码混杂在一起，导致以下问题：

1. **源码目录污染**：难以区分哪些是项目原创代码，哪些是外部参考资料
2. **版本库膨胀**：大型参考资料文件导致 Git 仓库体积过大
3. **部署冗余**：参考资料随项目部署，增加分发成本
4. **维护困难**：参考资料更新时难以追踪和管理

通过建立独立的 reference/ 目录，可以系统性地解决上述问题。

### 大文件存储策略

对于大型参考资料，系统支持两种处理策略：

**策略一：Git LFS（Large File Storage）**

Git LFS 是 Git 的扩展，专门用于管理大文件。当参考资料中存在较大的二进制文件（如数据集、预训练模型、媒体文件等）时，可以将文件内容存储在 LFS 服务器上，仓库中仅保留轻量的指针引用。这样可以保持仓库的紧凑性，同时保留完整的版本控制能力。

**策略二：外部链接引用**

如果参考资料已经存在于外部存储（如云存储服务、官方网站、学术数据库等），建议仅在 reference/ 目录中保留指向这些资源的链接（URL 或下载说明），而不是将文件本身复制到仓库中。这种方式可以彻底避免仓库膨胀问题，但需要确保链接的长期可用性。

### 版本控制集成

reference/ 目录可以纳入 Git 版本控制，但需要根据内容类型做出合理决策：

- **小型文本资料**（论文摘要、博客摘录）：直接纳入版本控制，与项目代码一起管理
- **大型二进制资料**：使用 Git LFS 或保持外部链接，避免污染仓库历史

## 使用场景

### 适用场景

1. **技术调研阶段**：团队在引入新技术前，需要参考大量外部资料进行技术选型
2. **学习复现阶段**：参考开源项目或论文实现时，存放原始参考资料
3. **竞品分析阶段**：收集和整理对标项目的技术文档和代码
4. **知识沉淀阶段**：将团队学习过程中收集的优质资料集中管理

### 优势

| 优势 | 说明 |
|------|------|
| 目录清晰 | 明确区分源码与参考资料，提升项目可读性 |
| 便于复用 | 团队成员可以快速找到历史积累的参考资料 |
| 灵活部署 | CI/CD 流程可以灵活选择是否包含该目录 |
| 责任明确 | 参考资料的作者归属清晰，不与项目代码混淆 |

### 注意事项

1. **保持只读属性**：reference/ 目录中的内容应保持原始状态，不应被直接修改
2. **定期维护**：检查外部链接的有效性，及时移除失效资源
3. **明确来源**：建议为每份参考资料添加简短的说明文件，标注来源和用途
4. **避免重复**：不应将 reference/ 作为通用的文件存储仓库，只存放真正具有参考价值的资料

## 原文证据

根据原始资料描述：

> "可包含 clone 的开源项目、论文、技术博客摘录等。"

> "大文件建议使用 Git LFS 或仅保留链接。"

这些原文直接说明了 reference/ 目录的定位、内容边界和管理策略。
