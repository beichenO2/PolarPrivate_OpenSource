# 项目参考资料的目录治理规范

## 概述

在软件项目的开发过程中，团队通常需要参考大量的外部资料，包括开源项目、学术论文、技术博客、官方文档等。这些资料对于理解技术背景、学习先进实践、对比解决方案具有重要价值。然而，如果这些参考资料随意放置在项目代码中，会造成源码目录混乱、版本库膨胀、部署包冗余等问题。

为此，一个良好的实践是在项目根目录下建立专门的 **reference/** 目录，用于集中存放只读参考资料。该目录的设计原则是将外部参考资料与项目自身源代码严格隔离，确保源码目录的纯粹性，同时为团队提供一个集中查阅参考材料的统一入口。

该目录的核心定位是“只读”——即存放的是外部来源的引用材料，而非项目自身的代码实现。这种定位明确了 [[entity-reference-directory|reference/ 目录]]的属性边界，有助于团队成员快速识别哪些是原创代码，哪些是外部参考。

## 核心要点

### 目录定位

- **只读属性**：[[entity-reference-directory|reference/ 目录]]中存放的所有内容均为只读参考资料，不包含项目源代码
- **内容边界**：可存放的内容类型包括但不限于：
  - Clone 的开源项目（完整或部分）
  - 学术论文和技术规范文档
  - 技术博客文章摘录或全文
  - SOTA（State of the Art）技术对标材料
  - 外部工具的使用手册或教程

### 存储策略

大文件管理是参考资料存储中的常见挑战。当参考资料包含较大的二进制文件（如数据集、预训练模型、视频素材等）时，需要采用以下策略：

- **Git LFS 方案**：对于需要版本管理的大文件，推荐使用 Git Large File Storage (LFS) 扩展，将大文件内容存储在独立的 LFS 服务器上，仅在仓库中保留轻量引用
- **链接引用方案**：如果大文件已经存在于外部存储（如云存储、官方网站下载链接等），建议仅保留链接地址，而非将文件本身纳入仓库

### 目录隔离原则

该规范的核心价值在于通过目录隔离实现以下目标：

1. **源码纯净性**：确保 src/ 或 lib/ 等核心代码目录不混入外部参考资料
2. **职责分离**：明确区分"我们写的代码"和"我们参考的材料"
3. **便于维护**：参考资料可以独立更新，不影响项目核心代码的版本历史

## 知识结构

### 目的

解决项目中外部参考资料与源代码混杂导致的目录混乱问题，同时避免大文件导致仓库体积膨胀和克隆效率低下。

### 方法

在项目根目录建立专门的 [[entity-reference-directory|reference/ 目录]]，通过命名和结构规范集中管理所有外部参考资料。对于不同类型的参考资料和不同大小的文件，采用差异化的存储策略。

### 原理

通过目录级别的物理隔离实现关注点分离（Separation of Concerns），使得：

- 开发者可以快速定位所需的技术参考资料
- CI/CD 流程可以灵活选择是否包含 [[entity-reference-directory|reference/ 目录]]（如构建镜像时排除）
- Git 仓库可以根据需要仅克隆源码而不包含大型参考资料
- 代码审查时能够明确区分"参考材料引用"与"原创代码实现"

## 待探讨

在实际应用本规范时，可能需要进一步明确以下问题：

### 边界划定

- **reference/ 与 doc/ 的边界**：项目自身编写的文档（如 API 文档、开发指南）与外部参考资料之间的界限如何划分
- **参考资料的时效性**：外部参考资料可能随时间过时，是否需要建立资料的有效期或更新机制

### 版本管理

- **参考资料版本锁定**：当外部资料更新时，是否需要在 [[entity-reference-directory|reference/ 目录]]中记录引用的具体版本号
- **变更追溯**：参考资料本身的变更是否需要像代码一样记录变更日志

### 团队协作

- **访问权限控制**：不同角色（开发、测试、产品）是否需要对 [[entity-reference-directory|reference/ 目录]]有不同的访问级别
- **贡献指南**：团队成员是否可以向 [[entity-reference-directory|reference/ 目录]]添加新的参考资料，需要何种审核流程

### 实践扩展

- **自动化更新**：是否需要建立机制定期检查外部参考链接的有效性
- **搜索增强**：如何让 [[entity-reference-directory|reference/ 目录]]中的资料更易于被团队成员发现和检索
